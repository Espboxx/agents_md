# Constraints

## 技术约束
- 运行环境：面向 Windows PowerShell / PowerShell CLI，本地文件系统必须可写，且需要可解析 `$env:USERPROFILE`。
- 语言 / 版本：仓库主体是 PowerShell 脚本，核心实现依赖 `Set-StrictMode -Version Latest`、参数块、`Read-Host` 交互、`.NET` 文件 API 与 `Get-Date` 时间戳。
- 依赖限制：不依赖数据库、Web 服务或第三方模块；唯一可选的额外工具是 `PSScriptAnalyzer`，仅用于静态检查。
- 文件编码：所有由脚本写出的目标文件都应保持 UTF-8 无 BOM，与 `Write-Utf8File` 的实现一致。

## 业务约束
- 业务边界：仓库职责是安装和初始化 harness 基础设施，不负责业务系统部署、代码生成之外的运行时服务或数据持久化。
- 输出边界：脚本只写本地文件，包括用户目录下的全局配置和指定项目目录下的文档/`AGENTS.md`；不会访问远端 API。
- 审计与回滚：对已有文件的覆盖或追加必须保留 `.bak.<timestamp>` 备份，保证用户可回退到改写前状态。

## 工程约束
- 不允许的改动类型：非必要不要引入新依赖、不要改变现有公开参数 `-RepoRoot` / `-Force` 的语义、不要把当前脚本扩展成与本仓库目标无关的大型框架。
- 必须保留的兼容性：继续支持交互式确认、当前目录安装、手工输入项目目录、已有 `AGENTS.md` 的追加/覆盖分支，以及未变化文件直接跳过写入的行为。
- 稳定性要求：路径解析必须明确，避免把全局文件写到错误目录；写文件前必须确保目录存在；错误应在 `Stop` 模式下尽早暴露。
- 用户体验要求：交互提示和安装结果输出需要清晰可读，用户应能从 `[WRITE]`、`[APPEND]`、`[BACKUP]`、`[SKIP]` 日志判断脚本实际做了什么。

## 交付约束
- 时间约束：仓库没有构建流水线或测试矩阵，验证以最相关、最小化的本地检查为主。
- 验收约束：至少通过 PowerShell Parser 语法检查；若改动涉及文档模板或写入逻辑，还应确认改动范围准确、目标文档无模板占位残留。
